Or does it...

Many (most?) Grunt plugins are simply wrappers around other tools.

Those tools are perfectly functional on their own without Grunt.

One example: Grunt Karma

Karma is a NodeJS based test runner.

Grunt Karma is a Grunt plugin for Karma.

# Start Karma from the command line using your config:
$ karma start karma.conf.js

The cost of using Grunt plugins instead of calling the underlying tool directly

  • Two sources for documentation
  • When something fails, you have two things to investigate
  • Potentially slower to receive updates
  • An additional dependency to install
  • Grunt plugin might require a specific old version of Grunt
  • Too much magic; less visibility into how things work; harder for us to truly understand our tools
  • Too much text on this slide

When adding a tool to your project

Consider implementing the tool in the simplest possible way, with as few dependencies as possible.

Only add layers like Grunt/Gulp/Webpack if they actually add value.

No. Actually, it runs on:

Is the abstraction that Grunt gives us really worth the cost of having an intermediary between us and our tools?

Grunt/Gulp/Webpack - not the only way

We can configure sequences of tasks with a tool that we all already have:

Scripts in package.json

"devDependencies": {
  "karma": "0.13.22"
},
"scripts": {
  "test": "karma start karma.conf.js"
}

Now in the command line we can run npm run test. This will execute the script we have defined in test, starting Karma and passing in any arguments we have defined.

Normal command line scripts

npm scripts can run anything you can run on your command line, e.g:

"scripts": {
  "clean": "rm -rf build/*"
}

But don‚Äôt forget our framework supports Windows.

Some Bash commands like rm (remove) and mv (move/rename) are not available on cmd.exe.

Cross platform commands

"devDependencies": {
  "rimraf": "2.5.2"
},
"scripts": {
  "clean": "rimraf build/*"
}

Performs the rm -rf task but also works on Windows.

Sequential tasks

"scripts": {
  "precss": "sass-lint style.scss.css",
  "css": "node-sass style.scss.css style.css"
}

If you run npm run css, npm will run the precss task first, and only proceed if this task is successful.

More sequential tasks

You can use the && operator

"scripts": {
  "css": "sass-lint style.scss.css && node-sass style.scss.css style.css && postcss --use autoprefixer style.css --replace && cleancss style.css"
}

Lint, then run Sass, then add prefixes, then minify.

But having more than one operation on a single line makes your script complex and hard to read üôÅ

More sequential tasks

"devDependencies": {
  "npm-run-all": "2.1.1"
},
"scripts": {
  "lint": "sass-lint style.scss.css",
  "sass": "node-sass style.scss.css style.css",
  "prefix": "postcss --use autoprefixer style.css",
  "minify": "cleancss style.css",
  "css": "npm-run-all lint sass prefix minify"
}

Much easier to read üôÇ

More sequential tasks

"devDependencies": {
  "npm-run-all": "2.1.1"
},
"scripts": {
  "css:lint": "sass-lint style.scss.css",
  "css:sass": "node-sass style.scss.css style.css",
  "css:prefix": "postcss --use autoprefixer style.css",
  "css:minify": "cleancss style.css",
  "css": "npm-run-all css:*"
}

Even easier to read üòÄ

Parallel/concurrent tasks

"devDependencies": {
  "npm-run-all": "2.1.1"
},
"scripts": {
  "middleman": "bundle exec middleman",
  "sass": "node-sass style.scss.css style.css --watch",
  "serve": "npm-run-all --parallel middleman sass"
}

Middleman and LibSass will run at the same time.

Middleman will watch ERB files and LibSass will watch SCSS files.

Streaming / pipelines

  • Streaming the output from one task into the input of the next task is one of the reasons why GulpJS is so popular.
  • Allows sequential tasks to run efficiently because it doesn‚Äôt have to save to the file system in between each task.
  • But of course GulpJS didn‚Äôt invent streaming. We can also do it on the command line with the | (pipe) operator, which means we can do it with npm scripts.

Streaming

"scripts": {
  "css": "node-sass example/style.scss.css | postcss --use autoprefixer | cleancss -o example/style.css"
}

Streaming from LibSass to PostCSS (for auto-prefixing)
to Clean-CSS (for minification).

Nice, but again, hard to read üôÅ

The power of Node

Not all npm modules expose a command line interface
(e.g. directory-colorfy and directory-encoder).

Sometimes you want more control and flexibility.

Write custom JavaScript and run it with Node from your npm scripts!

directory-colorfy.js (plain JavaScript)
var dc = require('directory-colorfy'),
    input = 'svgs/svgs-temp',
    output = 'svgs/svgs-temp',
    options = {};
    
new dc(input, output, options).convert();
package.json (tells NodeJS to run the JavaScript)
"scripts": {
  "svg:colorfy": "node directory-colorfy.js"
}
